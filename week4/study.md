# 📘 Today I Learned

### 1. 오늘 배운 내용

- fetch()로 외부 API에 HTTP GET 요청을 보내고, 응답 데이터를 받아 화면에 반영하는 전체 흐름 구현
- Promise .then()/.catch()와 async/await를 사용한 비동기 코드 작성 -비동기 요청 상태(로딩/성공/실패/재시도)에 따라 UI를 다르게 표시하는 구조 구현 -외부 API 응답 데이터를 화면에서 쓸 수 있는 형태로 변환하는 과정 학습
- localStorage로 데이터를 유지하고, 데이터 기준으로 화면을 갱신하는 구조 적용

### 2. 핵심 정리 (내 언어로)

- fetch(url)을 호출하면 요청이 나가고, JS는 응답을 기다리지 않고 다음 코드를 실행한다. 응답이 오면 그때 .then() 안의 코드가 실행되고, 그 결과로 화면이 바뀐다
- async/await는 비동기 코드를 위에서 아래로 순서대로 읽히게 해주지만, 실제로는 await 지점에서 멈추고 응답이 올 때까지 기다린다. 값이 전달되는 시점은 Promise가 resolve된 이후다
- API에서 받아온 데이터는 화면에서 바로 쓸 수 없어서, 필요한 필드만 골라내고 없는 값은 임의로 채워 새 객체로 변환했다. 이 과정이 없으면 데이터 구조가 달라 화면이 깨진다 -요청 중에 버튼을 비활성화하고 "불러오는 중..." 문구를 보여주는 이유는, 사용자가 현재 상태를 알 수 없으면 중복 요청을 보내거나 오류를 인지하지 못하기 때문이다
- DOM을 직접 조작하면 데이터와 화면이 따로 놀게 되는 문제가 생긴다. 데이터 배열을 기준으로 매번 화면을 다시 그리는 방식이 더 일관성 있고 관리하기 쉽다
- URL의 ?results=5&nat=kr 같은 쿼리 파라미터로 원하는 조건의 데이터를 요청할 수 있고, GET 메서드는 서버에서 데이터를 읽어오는 용도로 사용된다

### 4. 느낀 점

- 비동기 코드는 실행 순서가 눈에 보이는 것과 달라서 처음에 헷갈렸는데, 직접 짜보면서 언제 값이 들어오는지 감이 잡혔다
- 상태 관리를 제대로 안 하면 버튼이 중복 클릭되거나 이전 데이터가 남는 버그가 생긴다는 걸 실제로 겪으면서 배웠다
- 데이터를 기준으로 화면을 그리는 구조가 처음엔 번거로워 보였지만, 기능이 늘어날수록 훨씬 정리가 잘 된다는 걸 느꼈다
